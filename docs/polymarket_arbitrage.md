# Polymarket Arbitrage Math

A small, dependency-free library (`js/polymarket.js`) implementing the core
mathematics behind systematic prediction-market arbitrage. It is pure math —
no networking, no orders, no trading advice — extracted from:

- **"Unravelling the Probabilistic Forest: Arbitrage in Prediction Markets"** —
  arXiv:2508.03474
- **Bregman-divergence market making** — arXiv:1606.02825

Run the worked examples (which double as a smoke test):

```bash
node scripts/polymarket_arb_demo.js
```

In the browser it attaches to `window.BTC.polymarket` (loaded from `index.html`);
under Node it is a normal CommonJS module.

## The four ideas, and where they live

| Article idea | Function | What it is |
|---|---|---|
| **Strategy 1** — `YES + NO != $1` | `detectSingleConditionArbitrage` | A complete set (1 YES + 1 NO) always redeems for exactly $1. Buy the set under $1, or mint-and-sell it over $1. Fees and a minimum-profit threshold are parametric. |
| **Strategy 2** — market rebalancing | `detectPartitionArbitrage` | For *k* mutually-exclusive **and** exhaustive outcomes, exactly one YES pays $1. Buy every YES if they sum < $1; buy every NO if they sum > $1. Guaranteed edge = `|1 − Σ YES|`. This was the larger bucket in the paper (~$29M of $40M). |
| **The "probabilistic forest"** — no-arbitrage projection | `frankWolfeProject` + LMO builders | The honest, general case. |
| **Sizing & execution** | `kellySize`, `simulateSequentialFill` | Why detection isn't enough. |

## Why simple sum-checks aren't the real problem

For *n* logically-linked binary conditions, the arbitrage-free price vectors form
the **marginal polytope**

```
M = conv{ x_v : v is a logically valid joint outcome }
```

where `x_v ∈ {0,1}^n` marks which conditions are TRUE in outcome `v`. A quoted
price vector `q` admits a guaranteed-profit trade **iff `q ∉ M`**. The
minimum-cost trade that removes the arbitrage is the projection of `q` onto `M`
under the **Bregman divergence** generated by the market maker's cost function.
LMSR uses (negative) entropy, whose Bregman divergence is the generalized KL
divergence

```
D(p, q) = Σ_i [ p_i ln(p_i / q_i) − p_i + q_i ]
```

and the **dollar value of the guaranteed arbitrage equals `D(p*, q)`** at the
projection `p* = argmin_{p ∈ M} D(p, q)` — the paper's headline result,
implemented in `bregmanKL` / `frankWolfeProject`.

## Why Frank-Wolfe (you never enumerate 2ⁿ)

`M` has exponentially many vertices, so you cannot project directly. Frank-Wolfe
only needs a **linear-minimization oracle (LMO)**: given a gradient `g`, return
the single best vertex `argmin_{v ∈ M} ⟨g, v⟩`. Because vertices are 0/1 outcome
indicators, that oracle is just *"find the highest-value valid outcome"* — an
integer program (Gurobi in production; here a brute-force or a structured
partition-aware oracle). Each iteration adds one vertex to an **active set**, so
after ~100 iterations you track 100 vertices instead of 2ⁿ. Typical convergence
is well under 50 iterations for the examples here.

LMO builders provided:

- `makeEnumeratedLMO(validOutcomes)` — brute force over an explicit list (good
  for small `n`, or a pre-filtered candidate set such as the "AI dependency
  detection" output).
- `enumerateValidOutcomes(n, isValid)` — the 2ⁿ brute-force baseline Frank-Wolfe
  replaces (use only for `n ≲ 22`).
- `makePartitionLMO(n, groups)` — O(n), no enumeration, for one-hot partitions
  (e.g. "exactly one team wins"). This is the structured IP the paper exploits.

## Sizing and execution — why detection isn't enough

- **`kellySize`** — modified Kelly for a leg with *execution risk*. Pure arbitrage
  looks like infinite edge, so vanilla Kelly says "bet everything", ignoring that
  a leg can fail to fill and turn the trade directional. We model success
  probability and asymmetric gain/loss, then cap the size two ways the paper
  insists on: a bankroll fraction (half-Kelly default) and **≤ 50% of visible
  book depth** so you don't move the market against yourself.
- **`simulateSequentialFill`** — Polymarket is a Central Limit Order Book; legs
  fill **sequentially**, not atomically. The simulator walks each leg's book and
  returns the realized average price with slippage. The demo reproduces the
  article's trap: "buy YES @ 0.30, buy NO @ 0.30" becomes "0.30 then 0.85" once
  the first leg moves the market, turning a planned **+$400** into a realized
  **−$122.50**.

## API summary

```js
const pm = require('./js/polymarket.js'); // or window.BTC.polymarket

pm.detectSingleConditionArbitrage(yesAsk, noAsk, { feeRate, minProfit, yesBid, noBid })
pm.detectPartitionArbitrage(yesPrices, { feeRate, minProfit })
pm.scanPartitionGroup(legs, opts)                 // detection + Bregman value in one call

pm.bregmanKL(p, q)                                // generalized-KL divergence ($ value)
pm.frankWolfeProject(q, lmo, { maxIter, tol, lineSearch })
pm.makeEnumeratedLMO(validOutcomes)
pm.makePartitionLMO(n, groups)
pm.enumerateValidOutcomes(n, isValid)

pm.kellySize({ bankroll, gain, loss, pSuccess, kellyFraction, bookDepthShares, legPrice, depthFraction })
pm.simulateSequentialFill(legs)
```

Everything is a pure function. None of it places an order or talks to a network.
